# RECOVERY

## 기본 원칙

검증이 실패하면 곧바로 광범위한 수정에 들어가지 않는다. 진단부터 한다.

## 실패 루프

검증 단계가 실패하면 다음 순서를 따른다.

1. 실패 출력을 끝까지 읽는다.
2. 실패 유형을 분류한다: 구현 버그 / 잘못된 테스트 / 의존성 누락 / 환경 문제 / 요구사항 불명확 / scope 충돌 / mock 경계 위반.
3. PRD.md, PLAN.md, VALIDATION.md, 02_DATA_MODEL.md와 비교한다.
4. 가장 작은 가역 수정 1회를 한다.
5. 가장 작은 관련 검증부터 재실행한다.
6. PROGRESS.md를 업데이트한다.

## 재시도 한계

같은 검증이 서로 다른 3회 시도(3 attempts) 후에도 실패하면:

골이 진단·검토 모드로 전환되고 자체 수정을 멈춘다. Claude Code는 `/goal pause`를 지원하지 않으므로 사람의 결정을 기다린다. 필요하면 `/goal clear` 후 재설정한다.

다음을 보고한다:
- 실패한 명령 또는 기준
- 시도한 3가지 수정
- 각각의 실패 이유
- 다음으로 안전한 옵션
- 사용자·제품 결정이 필요한지 여부

## scope 잠금

다음은 금지한다:
- 무관한 모듈을 다시 작성
- PLAN.md에 명시되지 않은 public API 변경
- 02_DATA_MODEL.md에 명시되지 않은 DB 스키마 변경 (마이그레이션으로 명시할 것)
- 검증을 통과시키기 위해 테스트를 삭제·skip
- 이해 없이 에러를 침묵 처리
- 좁은 문제를 광범위한 refactor로 해결
- 검증 명령 자체를 다른 명령으로 교체
- 문서·롤백 경로 없는 feature flag 추가
- **mock이어야 할 경로를 실 외부 API로 연결** (PG 결제·알림 발송·AI 챗봇·AI OCR·판례 AI는 mock 어댑터로만 — 실 키/실 호출 추가 금지)
- **실로직이어야 할 경로를 mock·하드코딩으로 우회** (Settlement 룰베이스 산출·합의 상태전이·CRM 알림 룰·이슈보드·커뮤니티/투표·보수업체 매칭은 실제 동작해야 함)
- 정산 분담비율을 룰엔진 대신 임의 AI 추론으로 산출 (LH기준·내구연수·감가상각이 본체, 판례 AI는 mock 보조)
- 임대인이 임차인 보증금을 양측 합의 없이 앱이 자동 공제
- Phase 1 기능(대시보드/점검/HouseLog/구독) 회귀로 깨뜨리기
- 주민등록번호 등 고유식별정보를 DB·로그·S3·OCR 결과에 평문 저장·노출 (PASS는 `verified_at`만, OCR은 마스킹·비저장)
- 백엔드 테스트만 그린으로 두고 UI 클릭 완주 없이 마일스톤 완료 처리

## 방향 재확인 규칙

접근법을 바꾸기 전에:

1. 골 문장을 다시 읽는다.
2. PRD.md(01_PRD.md §6 안 만드는 것)와 03_PHASES.md Phase 2/3 범위를 다시 읽는다.
3. PLAN.md의 현재 마일스톤과 mock-first 원칙을 다시 읽는다.
4. 다음 수정이 현재 마일스톤에 직접 기여하는지, mock/실로직 경계를 지키는지 확인한다.
5. 04_PROJECT_SPEC.md의 "절대 하지 마" 항목과 충돌하지 않는지 확인한다.

## 되돌리기 규칙

자기 자신의 마지막 실패 변경만 되돌린다. 단, 다음 중 하나일 때:
- 변경이 검증을 더 악화시켰을 때
- 무관한 변경을 함께 들여왔을 때
- PRD / PLAN과 충돌할 때
- mock/실로직 경계를 위반했을 때

사용자가 수동으로 만든 변경은 명시적 지시 없이 절대 되돌리지 않는다.
