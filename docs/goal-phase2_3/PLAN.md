# PLAN — 버틀러 Phase 2 + 3 (mock-first)

## 목표

Phase 1이 완성된 버틀러 위에, **수선비 정산·임대차 CRM·이슈보드·커뮤니티·보수업체 매칭의 핵심 비즈니스 로직을 실제로 동작**시키고, **외부 의존(PG 결제·알림 발송·AI 챗봇·AI OCR·판례 AI)은 전부 mock 어댑터로 대체**해, 임차인을 포함한 전 역할이 웹/모바일에서 **정산→합의→(mock)결제, 수선요청→이슈보드, 커뮤니티·투표, 보수업체 매칭, AI 보조(mock)**까지 클릭으로 완주할 수 있는 Phase 2+3 제품을 완성한다.

## mock-first 원칙 (전 마일스톤 공통)

- **mock 어댑터(실동작 X, 가짜 응답)**: PG 결제, 알림 발송(카카오 알림톡/SMS), AI 챗봇(RAG), AI OCR 등기부, 판례 AI.
- **실제 로직 동작(mock 아님)**: Settlement 룰베이스 산출, 양측 합의 상태전이, CRM 알림 룰 산출, 수선요청 이슈보드, 커뮤니티/투표, 보수업체 매칭.
- 외부 의존은 인터페이스(어댑터)로 추상화하고 mock 구현을 기본값으로 둔다. 실 키가 들어오면 실 어댑터로 교체만 하면 되도록 셸을 남긴다.
- mock이라도 **UI에서 클릭으로 흐름을 완주**할 수 있어야 마일스톤 완료다 (백엔드 테스트 그린만으로 완료 처리 금지).

## 마일스톤 1: Lease·TENANT 활성화 + 수선요청 이슈 협업보드

- 범위(Scope): Lease 엔티티 활성화(스키마 마이그레이션 명시), TENANT role 추가(packages/shared enum + RBAC + 웹앱 임차인 진입 화면·메뉴 분기), 임차인 초대/연결 플로우(초대 토큰), MaintenanceRequest 엔티티 + 수선요청 이슈 협업보드(임대인·임차인·관리자 공동, 사진 첨부·진행상태 전이·코멘트 이력 아카이빙), 수선요청 발생 시 HouseLogEntry append 연결.
- 완료 조건: 임차인이 가입→Lease로 임대인 물건에 연결→웹에서 수선요청 생성→임대인/관리자가 보드에서 상태전이(접수→처리중→완료)→이력이 남고 HouseLog에 기록. 타 임차인/타 물건 접근 403. UI 클릭 완주.
- 검증: `npm test -- lease maintenance` + 임차인·임대인 양측 웹 수동 완주 + RBAC 403 확인 + 임차인 role이 Phase 1 흐름을 깨지 않는 회귀.

## 마일스톤 2: 수선비 정산 엔진(룰베이스) + 양측 합의

- 범위(Scope): Settlement 엔티티, 룰베이스 정산 엔진(ai-python — LH 부담기준표 + 표준 내구연수 + 감가상각), **InspectionItem 등급·내구연수·marked_defect를 실제 참조**해 임대인/임차인 분담비율 산출, 산출 근거 스냅샷 보관(기준·내구연수·감가상각 계산식 동결), 정산 리포트 생성, 임대인·임차인 양측 합의 상태전이(제안→이의/조정→합의 또는 결렬). 보증금 자동공제 금지(합의 후 결제로 분리).
- 완료 조건: MOVE_OUT 점검 데이터 기준으로 분담비율이 룰대로 산출되고 근거가 리포트에 첨부됨. 임대인·임차인이 웹에서 합의 상태를 전이해 합의 완료까지 도달. AI 임의추론으로 분담비율을 만들지 않음(룰엔진 단위 테스트로 보장). UI 클릭 완주.
- 검증: `pytest tests/settlement` (LH기준·감가상각 경계 케이스) + `npm test -- settlement` + 점검 데이터→정산→양측 합의 통합 테스트 + 수동 완주.

## 마일스톤 3: 임대차 CRM + 자동알림 + (mock)PG 결제

- 범위(Scope): 임대차 CRM(계약·물건·임차인 운영 뷰), 자동알림 룰 산출(계약만료 D-Day / 월세미납 / 수선요청 / 리포트 도착 / 정산 합의요청), Notification 엔티티, **알림 발송은 mock 어댑터**(카카오 알림톡/SMS 미연동 — 발송 로그만 적재, 인앱 알림센터 노출), Payment 엔티티 + **mock PG 어댑터**(구독료 + 정산금), 정산 합의 완료→(mock)결제→Payment 기록→HouseLog/Notification 연동까지 닫기.
- 완료 조건: 알림 룰이 실제 데이터(계약 end_at, 미납, 요청)로 트리거되어 인앱 알림센터에 노출(발송은 mock). 정산→합의→mock결제까지 임대인·임차인 웹에서 클릭 완주, Payment status가 정확히 전이. 실 결제는 일어나지 않음(mock 어댑터임이 코드·UI에 명시).
- 검증: `npm test -- crm notification payment` + 알림 룰 트리거 단위 테스트 + 정산 합의→mock결제 통합 테스트 + 수동 완주(알림센터·결제 완료 화면).

## 마일스톤 4: 단지 커뮤니티 + 전자투표 + 보수업체 매칭

- 범위(Scope): Community/Post/Vote 엔티티, 실소유주 인증 게이트(해당 단지 Property 소유/거주 검증)로 폐쇄형 가입, 단지 뉴스 피드·게시·댓글, 전자투표 생성·투표·집계, Vendor 엔티티 + 보수업체 디렉토리·평점·검색, 수선요청에서 업체 연결(선택). 전부 실제 로직.
- 완료 조건: 인증된 실소유주만 단지 커뮤니티 진입·게시·투표 가능(미인증 차단). 투표 생성→참여→집계 결과 노출. 보수업체 디렉토리 검색·평점 노출·수선요청 연결. UI 클릭 완주.
- 검증: `npm test -- community vote vendor` + 실소유주 게이트 미인증 차단 테스트 + 투표 집계 정확성 + 수동 완주.

## 마일스톤 5: AI 보조 전부 mock — 챗봇 · OCR 등기부 · 판례

- 범위(Scope): ChatbotLog 엔티티 + **mock RAG 챗봇 어댑터**(임대차법/세무 상담 — 고정/규칙 기반 가짜 응답, 외부 모델 미호출), OcrDocument 엔티티 + **mock AI OCR 어댑터**(등기부 권리관계 파싱·안전등급/깡통전세 진단 stub, **주민번호 마스킹·비저장 강제**), 룰베이스 정산에 붙는 **판례 AI 보조 mock**(유사 분쟁사례 안내 가짜 응답). 전부 어댑터 인터페이스 + mock 구현 + UI 노출.
- 완료 조건: 챗봇 질의→mock 답변, 등기부 업로드→mock 파싱 결과·안전등급 노출(주민번호 마스킹), 정산 화면에 판례 보조 mock 안내 노출까지 웹에서 클릭 완주. mock임이 코드·UI에 명시되고 실 모델 키 자리만 비워둠. OCR 결과에 주민번호 평문 0건.
- 검증: `npm test -- chatbot ocr` + `pytest tests/ocr_mock` + OCR 결과 주민번호 grep 0건 + 수동 완주(챗봇·등기부·판례 안내).
