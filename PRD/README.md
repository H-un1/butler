## 핵심 결정 요약

- **성격**: 산학협력 기반 실제 출시 제품 (풀스코프, 단기 MVP 아님)
- **핵심 고객**: 원격 임대인 / **다면 플랫폼**(임대인·임차인·점검자·관리자, +Phase3 보수업체)
- **정산 엔진**: 룰베이스(LH기준+감가상각) + AI 보조 (판례학습 AI 직접구축은 후순위)
- **점검 운영**: 전문 점검자 대행 파견
- **임차인**: 전용 앱/계정 (양면)
- **정산 결제**: 앱 내 결제(PG) — 단 보증금 자동공제 아닌 양측 합의 후
- **과금**: 보유 물건수 기준 + 구간 할인
- **스택**: React Native + React + Node.js + Python/FastAPI + MySQL/Redis + AWS
- **인증**: 소셜 로그인 + PASS 본인인증 (주민번호 비저장)

## Phase 요약

- **Phase 1 (MVP)**: 인증 · 물건등록+공공데이터 대시보드 · House Log · 스마트 점검 리포트 · 구독과금
- **Phase 2**: 수선비 정산(룰베이스) · CRM/알림 · 이슈보드 · PG 결제
- **Phase 3**: AI 챗봇 · 커뮤니티/투표 · AI OCR 등기부 · 보수업체 매칭

## 미결 사항 (NEEDS CLARIFICATION) 종합

1. 공공데이터 API(국토부/건축물대장/K-APT) 실제 가용성·요금·호출제한·실시간성
2. PASS 본인인증 대행사 선정·비용, 등기상 소유자와 인증명의 일치 검증법
3. 점검자 풀 확보 운영 모델 (자체고용/제휴/프리랜서)
4. AMI(자산관리지수) 산출 공식·가중치 정의
5. 구독 과금 구간 정의 (채당 단가, 구간 경계, 무료체험)
6. House Log 데이터 소유권·보관·파기 정책 (매도 시 귀속)
7. 점검 리포트 전자서명(법적 효력) 도입 여부
8. AI 결함분석/AMI 모델 자체학습 vs 외부 API
9. PG사 선정(Phase 2)·정산 수수료 정책
10. 모노레포 vs 멀티레포 구조 결정
